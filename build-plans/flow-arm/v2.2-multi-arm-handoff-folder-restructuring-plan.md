# v2.2 - Multi-Arm Handoff Folder Restructuring Plan (Post-V1 Enhancement)

## Status

**Post-V1 (v2) enhancement. NOT required for V1 completion.** This plan is not part of the original 6-chunk build (plans 01–08); the `v2.2` prefix marks it as an explicitly-separate "version 2" extension. **V1 ships and stays on the flat `SeriesHandoff/` layout** until this plan executes. Do not execute this plan if Flow Arm remains the only Arm — it adds structural overhead with no benefit until a second Arm is real, and Aaron has explicitly authorized cutover.

## Purpose

Restructure the shared handoff folder so each Arm has its own filesystem-isolated subtree. Each Arm's container mounts only its own subtree. Brain dispatches into the right Arm's subtree. Cross-Arm visibility becomes physically impossible at the filesystem layer rather than depending on code-discipline filtering.

## Background and motivation

V1 Flow Arm uses a flat `SeriesHandoff/` shared with Brain via Syncthing:

```text
~/HermesArms/flowarm/SeriesHandoff/
├── archive/
├── flowarm-status/
├── jobs/
│   ├── outgoing/
│   ├── claimed/
│   ├── completed/
│   ├── failed/
│   └── status/
└── results/incoming/
```

Plan 02 mounts this root into the Flow Arm container as `/workspace/handoff/`. There is one Arm, so the flat structure works.

When a second Arm (Mid Journey Arm or other) is added, the flat structure breaks:

- **Cross-Arm leakage:** Brain would dispatch all packets into one shared `outgoing/`, and each Arm would need code-discipline filtering (e.g., a top-level `arm` field in the packet) to know which packets are theirs. A bug in that filter could let an Arm read another Arm's packet.
- **Status namespace collision:** `flowarm-status/` is already Flow-Arm-specific by name, but generalizing to multiple Arms means multiple `<arm>-status/` directories all visible to all Arms.
- **Result-manifest path conflicts:** `results/incoming/<job_id>/manifest.json` could be written by either Arm if `job_id` namespaces aren't enforced.
- **Defense-in-depth lost:** SOUL.md + AGENTS.md provide policy-layer isolation; mount-layer isolation is the second line of defense (proven in Chunk 3 — the agent refused to even attempt `/workspace/series-vault` access). The flat structure puts the second line of defense entirely on Brain's dispatch correctness, not on the filesystem.

The right architectural posture is **per-Arm subtrees with per-Arm mounts**. Aaron flagged this during Chunk 4 review as a structural point worth capturing.

## Target Architecture

Post-restructure, the shared root holds per-Arm subtrees:

```text
~/HermesArms/flowarm/SeriesHandoff/
├── FlowArmHandoff/
│   ├── flowarm-status/
│   ├── jobs/
│   │   ├── outgoing/
│   │   ├── claimed/
│   │   ├── completed/
│   │   ├── failed/
│   │   └── status/
│   └── results/incoming/
├── MidJourneyHandoff/
│   ├── midjourney-status/
│   ├── jobs/{outgoing,claimed,completed,failed,status}/
│   └── results/incoming/
└── _shared/
    └── archive/         # historical packets across all Arms, post-completion
```

Each Arm's container mounts ONLY its own subtree:

- Flow Arm: `~/HermesArms/flowarm/SeriesHandoff/FlowArmHandoff:/workspace/handoff`
- Mid Journey Arm: `~/HermesArms/midjourneyarm/SeriesHandoff/MidJourneyHandoff:/workspace/handoff` (the host-side path may differ if each Arm runs on its own device — Syncthing handles cross-device replication)

In-container, each Arm sees `/workspace/handoff/` with the same structure it sees today. The change is invisible from the agent's point of view.

## Exact Later Execution Steps

### Phase 1 — Prepare new layout alongside old

While V1 Flow Arm continues running on the flat structure:

1. Create the new subfolder tree without touching the old one:

   ```bash
   ROOT=~/HermesArms/flowarm/SeriesHandoff
   mkdir -p "$ROOT/FlowArmHandoff/jobs/outgoing"
   mkdir -p "$ROOT/FlowArmHandoff/jobs/claimed"
   mkdir -p "$ROOT/FlowArmHandoff/jobs/completed"
   mkdir -p "$ROOT/FlowArmHandoff/jobs/failed"
   mkdir -p "$ROOT/FlowArmHandoff/jobs/status"
   mkdir -p "$ROOT/FlowArmHandoff/results/incoming"
   mkdir -p "$ROOT/FlowArmHandoff/flowarm-status"
   mkdir -p "$ROOT/_shared/archive"
   ```

2. Confirm Syncthing replicates the new subfolders to Brain side. Have Brain write a marker file under `SeriesHandoff/FlowArmHandoff/_sync-test-$(date +%s).txt`; confirm it appears on the Flow Arm device. Identical marker test for `_shared/`.

3. Verify the new subfolder tree is empty (no stale data) before any cutover.

### Phase 2 — Brain side: dispatch redirect

Brain's packet dispatcher writes to `SeriesHandoff/jobs/outgoing/<job_id>.json` today. Update Brain to write to `SeriesHandoff/FlowArmHandoff/jobs/outgoing/<job_id>.json` for any Flow-Arm-targeted job. Brain's status sidecar writer follows the same path: `SeriesHandoff/FlowArmHandoff/jobs/status/<job_id>.status.json`.

If Brain has a packet-routing layer that decides which Arm gets which packet, this is where the per-Arm subtree decision happens.

**Run Brain in the new path while Flow Arm is still on the old mount** for one full job cycle. Brain dispatches a test packet to the new location; Flow Arm doesn't pick it up (because it's still mounted on the old path); Aaron drains the test packet manually. This validates that Brain-side path changes work without affecting Flow Arm operations.

### Phase 3 — Flow Arm: mount cutover

Pick a no-job window (heartbeat status `idle`, no packet in `claimed/`).

1. Update `~/.hermes/profiles/flowarm/config.yaml` mount line:

   From:
   ```yaml
   mounts:
     - host_path: ~/HermesArms/flowarm/FlowArmWorkspace
       container_path: /workspace/flowarm
     - host_path: ~/HermesArms/flowarm/SeriesHandoff
       container_path: /workspace/handoff
   ```

   To:
   ```yaml
   mounts:
     - host_path: ~/HermesArms/flowarm/FlowArmWorkspace
       container_path: /workspace/flowarm
     - host_path: ~/HermesArms/flowarm/SeriesHandoff/FlowArmHandoff
       container_path: /workspace/handoff
   ```

2. Restart any Flow Arm Hermes session so the new mount takes effect. Container-side `/workspace/handoff/` view should be identical to before the cutover.

3. Update plan files and AGENTS.md to reflect the new host-side path:
   - `flow-arm-build-plans/01-device-prereqs-and-handoff-sync-plan.md` — directory creation steps reference `SeriesHandoff/FlowArmHandoff/...`
   - `flow-arm-build-plans/02-docker-hermes-flowarm-profile-plan.md` — mount specification updated
   - `flow-arm-build-plans/05-handoff-job-claim-and-result-manifest-plan.md` — host-side path examples updated; in-container paths unchanged
   - `AGENTS.md` — host-side comments updated; in-container references unchanged

4. **Container-side paths in AGENTS.md and Plan 05 do NOT change.** `/workspace/handoff/` still refers to the same logical structure; the mount underneath is what changed.

### Phase 4 — Validation

Inside `flowchat` after the cutover:

```text
List /workspace/handoff. List /workspace/handoff/jobs. Confirm structure matches what AGENTS.md describes. Then try to list /workspace/MidJourneyHandoff and /workspace/handoff/../MidJourneyHandoff — confirm both are not visible (no such file or directory).
```

Expected: Flow Arm sees its own subtree under `/workspace/handoff/`. **The isolation comes from what is mounted, not from `..` traversal:** `/workspace/handoff/..` does resolve to `/workspace/`, exactly as on any Linux filesystem. The reason `MidJourneyHandoff` is invisible is that the Flow Arm container's config.yaml mounts ONLY `~/HermesArms/flowarm/SeriesHandoff/FlowArmHandoff` to `/workspace/handoff` — there is no bind mount connecting any other host path to `/workspace/MidJourneyHandoff` or to any sibling location inside `/workspace/`. The container's filesystem is whatever the image provides plus the explicit bind mounts; nothing automatically connects sibling Arm subtrees on the host into this container.

In other words: **filesystem-level isolation here is "what's mounted, period."** Per-Arm subtrees on the host become per-Arm-isolated only because the per-Arm container mounts only one of them. If a future config.yaml accidentally added `~/HermesArms/flowarm/SeriesHandoff:/workspace/handoff-root` (the parent), Flow Arm would suddenly see all sibling Arms — that's exactly the misconfiguration this plan's mount discipline must prevent. Phase 4 validation should explicitly confirm only the per-Arm path is mounted.

Dispatch one tiny test packet from Brain through the new path. Flow Arm claims it, executes it (or, if no real generation is desired, generates a single placeholder image), writes manifest, returns it to handoff. Verify all paths land in the right per-Arm subtree.

### Phase 5 — Old layout cleanup

After one full successful job cycle through the new layout:

1. If the old `SeriesHandoff/jobs/`, `SeriesHandoff/results/`, and `SeriesHandoff/flowarm-status/` are empty (and they should be, since Brain stopped writing to them in Phase 2), archive them:

   ```bash
   mv ~/HermesArms/flowarm/SeriesHandoff/jobs ~/HermesArms/flowarm/SeriesHandoff/_shared/archive/legacy-flat-jobs-$(date +%Y%m%d)
   mv ~/HermesArms/flowarm/SeriesHandoff/results ~/HermesArms/flowarm/SeriesHandoff/_shared/archive/legacy-flat-results-$(date +%Y%m%d)
   mv ~/HermesArms/flowarm/SeriesHandoff/flowarm-status ~/HermesArms/flowarm/SeriesHandoff/_shared/archive/legacy-flat-flowarm-status-$(date +%Y%m%d)
   ```

2. Confirm Syncthing reflects the moves on Brain side.

### Phase 6 — Adding a second Arm (example: Mid Journey Arm)

This phase is template — actual second-Arm addition is its own build sequence (likely modeled on plans 01–07 with name substitutions).

1. On the Mid Journey Arm device, Syncthing syncs `~/HermesArmsXxx/midjourneyarm/SeriesHandoff/`. Within it, only `MidJourneyHandoff/` is the active subtree for that device.
2. Create the per-Arm subtree on the Mid Journey Arm device:
   ```bash
   ROOT=~/HermesArmsXxx/midjourneyarm/SeriesHandoff
   mkdir -p "$ROOT/MidJourneyHandoff/jobs/{outgoing,claimed,completed,failed,status}"
   mkdir -p "$ROOT/MidJourneyHandoff/results/incoming"
   mkdir -p "$ROOT/MidJourneyHandoff/midjourney-status"
   ```
3. Create `midjourneyarm` Hermes profile with mount `<...>/MidJourneyHandoff:/workspace/handoff` and a per-Arm SOUL.md / AGENTS.md.
4. Brain side learns the new dispatch path: Mid Journey jobs go to `MidJourneyHandoff/jobs/outgoing/`, status sidecars to `MidJourneyHandoff/jobs/status/`, results return to `MidJourneyHandoff/results/incoming/`.
5. Brain decides per-packet which Arm gets it (Flow Arm vs Mid Journey Arm) based on `target.platform` or similar packet field.

## Failure Handling

- **If Brain forgets to redirect dispatch (Phase 2):** packets pile up in the old `SeriesHandoff/jobs/outgoing/` while Flow Arm is on the new mount. Detection: heartbeat status doesn't transition to `claimed` despite a packet being in the old outgoing. Recovery: move the packets to the new path manually, re-dispatch via Discord claim.
- **If the new mount breaks Flow Arm visibility (Phase 3):** Flow Arm container starts but `/workspace/handoff/jobs/outgoing/` is empty / wrong / inaccessible. Roll back the config.yaml mount change to the old path. Investigate.
- **If a partial-migration state lingers** (some plans updated, some not): document the in-progress state in PROJECT_STATE.md so future sessions know what's complete. Do not paper over it.
- **If the trajectory-hermes profile shows ANY change** during the restructuring (sha256 drift, doctor failure, auth.json mtime change): stop immediately. trajectory-hermes is mounted from a different config and should not be affected by this plan; if it is, something is misconfigured at the Docker / Hermes layer.

## Explicit Do Not Do Notes

- Do not implement this plan before V1 ships and is running cleanly.
- Do not implement this plan if Flow Arm remains the only Arm. Structure for the case you have, not the case you imagine.
- Do not skip the Phase-1-then-Phase-2-then-Phase-3 sequencing. Mount cutover (Phase 3) before Brain dispatch redirect (Phase 2) creates a window where Flow Arm sees no packets while Brain still writes to the old path.
- Do not delete legacy directories in Phase 5 before confirming a full clean job cycle on the new layout.
- Do not put the `_shared/archive/` subfolder visible to per-Arm mounts. Each Arm should only mount its own per-Arm subtree, not anything labeled `_shared/`.
- Do not allow any Arm to write outside its own subtree. Plan 05's "Allowed and forbidden paths" section in AGENTS.md should be tightened per-Arm if needed.

## Notes For Brain side review

If you (Brain agent / Mac Studio reviewer) are reading this in advance of implementation:

- This plan does **not** change packet structure, manifest structure, status sidecar shape, or `packet_status` lifecycle. Plan 05 stays semantically identical at the contract level. Only the **paths** change on the host side.
- The single dispatch-side change is: `SeriesHandoff/jobs/outgoing/` → `SeriesHandoff/FlowArmHandoff/jobs/outgoing/`. Status sidecar writes follow the same prefix change.
- The single ingest-side change: read manifests from `SeriesHandoff/FlowArmHandoff/results/incoming/` instead of `SeriesHandoff/results/incoming/`.
- **Adding a new Arm in Brain's eyes** = adding a new dispatch path constant + per-Arm packet routing logic. No existing dispatch path moves; only new ones are added.
- **Backward compatibility:** if Brain ingests historical manifests from the legacy flat path post-restructure, they live under `SeriesHandoff/_shared/archive/legacy-flat-results-<DATE>/` after Phase 5. Brain ingest tooling may need to know about this archive location for retroactive lookups.
