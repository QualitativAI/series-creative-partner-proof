# v2.1 - Model Interchangeability Plan (Post-V1 Enhancement)

## Status

**Post-V1 (v2) enhancement. NOT required for V1 completion.** This plan is not part of the original 6-chunk build (plans 01–08); the `v2.1` prefix marks it as an explicitly-separate "version 2" extension. **V1 ships and stays as `nano-banana-pro` only** until this plan executes. Do not execute this plan until V1 has shipped at least one successful real job through the full Brain-dispatch → Flow-Arm-execute → Brain-ingest loop, and Aaron has explicitly authorized cutover.

## Purpose

Allow Brain-dispatched job packets to specify either `nano-banana-pro` or `nano-banana-2` as a `model` target on a per-prompt basis, and have Flow Arm execute against whichever model the packet specifies. The current V1 contract hardcodes `"model": "nano-banana-pro"` as the only Flow-Arm-executable target; this plan extends the whitelist while preserving every other invariant of the Plan 05 handoff contract.

## Background and motivation

- Discovered during Chunk 4 (2026-04-28) that Google rebranded the Flow image-generation product surface: "Nano Banana 2" is now the default for new projects, while "Nano Banana Pro" remains in the model dropdown alongside it (and Imagen 4). NBP appears to be the higher-quality / higher-latency option; NB2 appears to be the faster / cheaper option.
- For V1, NBP-only is the right scope — it minimizes contract surface and keeps the build deterministic.
- Post-V1, Aaron may want to A/B-test prompts across NBP and NB2, or batch-dispatch some prompts to NB2 (faster) and others to NBP (higher quality). That requires the packet to carry the model choice and Flow Arm to honor it.

## Exact Later Execution Steps

Both Brain side and Flow Arm side need coordinated changes. **Do not partial-deploy** — either both sides update together, or neither does. Brain emitting NB2 packets while Flow Arm only knows NBP would route NB2 prompts through the failure path. Flow Arm accepting NB2 while Brain only emits NBP is harmless but uninteresting.

### Step 1 — Brain side (Mac Studio): packet contract amendment

Update Brain's packet schema documentation to allow `model` ∈ `{"nano-banana-pro", "nano-banana-2"}` for any per-prompt target. Default if unspecified should remain `"nano-banana-pro"` to preserve V1 behavior.

Brain's ingest-side code that reads the manifest needs no change — manifests already carry the `model` field per Plan 05 step 5 — but Brain MUST verify that any tooling on the Brain side that filters or buckets results by model handles `nano-banana-2` as a known value rather than discarding it.

If Brain stores model-specific quality tiers, prompting playbooks, or canon-promotion rules, those need a parallel update so NB2 outputs are not dropped on the floor.

### Step 2 — Flow Arm side: Plan 05 amendment

In `flow-arm-build-plans/05-handoff-job-claim-and-result-manifest-plan.md`, change the packet interpretation rule (currently step 3 of "Exact Later Execution Steps"):

From:

```
- For Flow Arm, execute only targets where:
  ```json
  "model": "nano-banana-pro"
  ```
```

To:

```
- For Flow Arm, execute only targets where:
  ```json
  "model": "nano-banana-pro"
  ```
  or
  ```json
  "model": "nano-banana-2"
  ```

  Per-target rule: targets with any other `model` value are not Flow Arm's responsibility — skip them silently. This is the existing multi-target / multi-Arm silent-skip rule from Plan 05, extended from one recognized model to two.

  **Silent-skip is per-target only, NOT per-prompt or per-job:**
  - If a prompt has multiple targets and at least one is a supported model, execute the supported one and silently skip the unsupported ones (standard multi-target behavior).
  - If a prompt has zero supported targets across all its targets, that's a per-prompt error — record it in the manifest's `errors[]` array with `prompt_id` and `reason: "no supported model target"`. Per Plan 05 step 7's "no carve-out for partial success" rule, this routes the entire job through the post-claim failure path.
  - If the entire packet has zero supported targets across all prompts, that's a job-level failure per Plan 05's existing "If a packet has no nano-banana-pro target at all" rule — extended to "if a packet has no target with a model in the supported whitelist."

  **Brain-side validation responsibility (mandatory):** Brain MUST reject packets containing unrecognized model strings (typos, deprecated values, future models not yet whitelisted) at packet-validation time, BEFORE writing them to `outgoing/`. Flow Arm's silent-skip is correct under multi-Arm semantics — a target may legitimately belong to another Arm — but it cannot be the only line of defense against Brain-side dispatch bugs. The two layers together: Brain validates dispatch-time; Flow Arm silently skips per-target at execution-time; per-prompt and per-job failure rules surface "no supported work to do" cleanly.
```

### Step 5 — Flow Arm side: manifest preservation

The manifest's top-level `"model"` field per Plan 05 step 5 currently shows `"nano-banana-pro"` literally. Amend to record whichever model was actually used. If multiple prompts in the same job target different models, the manifest's top-level `"model"` field is no longer single-valued — possible resolutions:

- **(a)** Record `"model": "mixed"` and rely on per-prompt records to disambiguate. Plan 05 step 5 doesn't currently define a per-prompt `model` field in `prompts_completed[]`, so this requires a small schema extension.
- **(b)** Record the most-used model at the top level and add a per-prompt `model` field to each `prompts_completed[]` entry for full lineage.
- **(c)** Forbid mixed-model jobs at packet-validation time (Brain-side constraint) and keep manifests single-model.

Recommendation: **(b)**. Adds maximum lineage information with minimum contract surface. Brain can ingest single-model manifests as before (top-level `model` is canonical for those) and mixed-model manifests by reading per-prompt model tags.

## Failure Handling

- **If after Step 2 deploy the existing V1 NBP-only flow stops working,** immediately revert AGENTS.md and Plan 05 to single-model. Triage. The whitelist match must accept exactly `{nano-banana-pro, nano-banana-2}` — overly permissive matching is more dangerous than under-permissive.
- **If Brain emits a packet where ALL targets on a prompt are unsupported models** (no whitelist match anywhere on that prompt): per the per-prompt rule in Step 2, that's a per-prompt error. Add `errors[]` entry with `prompt_id` and `reason: "no supported model target for prompt"`. Per Plan 05 step 7's "no carve-out for partial success" rule (introduced in Chunk 2 round 3), this routes the entire job through the post-claim failure path — packet ends up in `failed/`, not `completed/`.
- **If Brain emits a packet where the entire packet has zero supported targets across all prompts:** that's a job-level failure. Mirror Plan 05's existing "no nano-banana-pro target at all" rule — extend it to "no whitelisted model target across any prompt." Post in `#flow-arm-log` with the failure reason; no execution attempted.
- **If a single target uses an unsupported model alongside a supported target on the same prompt:** the per-target silent-skip from Step 2 applies. Execute the supported target; the unsupported one is correctly assumed to belong to a different Arm under multi-Arm semantics. Brain-side validation (Step 1) is responsible for distinguishing "legitimate other-Arm target" from "typo / dispatch bug" before the packet leaves Brain.
- **If browser-harness can't locate either model label in the dropdown** (e.g., Google removed NBP from Aaron's tier, or renamed it again): Flow Arm posts a clear failure (`"Cannot find model 'Nano Banana Pro' in dropdown — Google may have changed product surface; please confirm tier/access"`), routes the packet through the post-claim failure path, and stays in `failed/` until Aaron resolves.

## Explicit Do Not Do Notes

- Do not implement this plan during the original 6-chunk build. It is a deliberate post-V1 enhancement.
- Do not weaken the per-target silent-skip rule for legitimate multi-Arm dispatch. Silent-skip per-target is correct; the dangerous case (no supported target on an entire prompt or an entire packet) has its own per-prompt and per-job failure rules.
- Do not rely solely on Flow Arm's silent-skip to catch Brain-side dispatch bugs. Brain-side packet validation (Step 1) is the load-bearing line of defense against typos / unrecognized model strings.
- Do not remove `nano-banana-pro` from the whitelist when adding `nano-banana-2`. Both must remain supported until and unless Google deprecates one.
- Do not change the filename convention as part of this plan. Filenames currently encode model name as a prefix (`nano-banana-pro_<job_id>_...`); for NB2, use `nano-banana-2_<job_id>_...`. This is a renaming-by-extension, not a contract change.
- Do not roll Brain side and Flow Arm side into a single review gate — they review separately, just sequenced (Brain first, then Flow Arm).
